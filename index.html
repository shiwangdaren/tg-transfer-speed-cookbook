Telegram 频道媒体搬运 · 上传下载提速 Cookbook

用 Go + gotd(MTProto user-account) 把一个/多个源频道的媒体高速搬运到目标频道，整套提速探索的工程复盘与可复用结论。
实测环境：单台 VPS（上行实测 ~210 Mbps，下行 ~1.45 Gbps）；~30 个 user 账号（含 4 个 Premium）；gotd v0.116.0。
TL;DR —— 如果你只看一段
  1. 真正的"下载慢"往往是你自己加的客户端限速中间件（ratelimit），不是 Telegram。先排查它。
  2. 单账号下载有 per-account（非 per-IP）的服务端限速：冷门文件 ~30-49 Mbps 封顶，连接/线程再多都撞墙。
  3. 突破单账号上限的两条路：① 多账号并行分块（把一个文件的块分给 N 个账号，origin 冷门源 ~463、热门源 ~706 Mbps）；② CDN 文件用 gotd "全 client" 下载器自动走 CDN（单 Premium 同 DC ~315 Mbps）。
  4. 上传用"连接池"（绕过 ratelimit、多连接）可达 ~145 Mbps，比单连接快百倍。
  5. 最终做一个下载调度联动器：自动用空闲账号、在"多账号并行 / 单会员 CDN"间选更快的那条。

0 · 背景与目标

系统要把源频道（公开或私有）里的媒体（视频/图片/文档）下载下来、可选加工后上传到多个目标频道。媒体常是几十~几百 MB 的视频，吞吐直接决定搬运速度。一开始单账号下载只有 ~1.6 Mbps，目标是榨干带宽。

关键前提：很多媒体可以用 "引用秒传"（messages.sendMedia 直接引用源文件的 file_reference）实现 0 下载 0 上传的服务端复制 —— 能用引用秒传就别下载。本文讨论的是必须真下载真上传（要重编码 / 改文件 / 源不可引用）的场景。

2 · 真凶：客户端 ratelimit 中间件

排查了半天 Telegram 限速、线程数、连接数，最后发现下载慢的主因是自己加的中间件。每个账号建 client 时加了：

tOpts.Middlewares = append(tOpts.Middlewares,
    floodwait.NewSimpleWaiter(),                 // 这个要留
    ratelimit.New(rate.Every(time.Minute/N), 3)) // N=rate_per_min；这个掐死了下载

分块下载里每个 1MB 的块 = 一次 upload.getFile RPC。当 rate_per_min=20 时，所有 API（含文件块）被限到 20 次/分钟 ≈ 20 MB/分钟 ≈ 2.7 Mbps，与文件大小无关。实测：

带 ratelimit(20/min)无 ratelimit倍数
下载1.6 Mbps27 Mbps17×
上传1.4 Mbps145 Mbps100×
结论：客户端节流应该只作用于"控制面"API（解析、发消息…），绝不能套在文件块 getFile/saveFilePart 上。批量传输要走一条没有 ratelimit 的连接（但保留 floodwait）。

3 · "线程越多越慢"的真相

gotd 的 downloader.WithThreads(n) 名字有误导：它是在同一条连接上开 n 个并发 getFile（parallel.go 确认），不是 n 条连接。所以：

  • 单连接上并发越多 → 越容易触发 per-connection / per-account 限速 → 越慢（实测 8 线程比 24 线程快）。
  • 要真正多连接，得用 gotd 的连接池：Client.Pool(max)（当前 DC，多连接）/ Client.DC(ctx,dc,max)（指定 DC，自动 exportAuthorization 转移）。
另一个老项目"单账号 300Mbps"的传说，拆开看其实是 单(借用)发送器 + N worker + 1MB 块 + 块级重试 + file_reference 刷新；它的代码注释自己写着「gigabit 的关键不是多客户端，而是处理 FileReferenceExpired」。在相同的冷门测试文件上 telethon 也只有 32 Mbps —— 没有魔法，差异全在文件是否走 CDN（见 §8）。

4 · 上传连接池 145Mbps

上传同理：单连接 + ratelimit = 1.4 Mbps。改用"当前 DC 连接池"（绕过 ratelimit、多连接）：

// tg.Client 上暴露一个无 ratelimit 的多连接调用器(保留 floodwait)
func (c *Client) HomePool(conns int) (telegram.CloseInvoker, error) {
    p, err := c.tg.Pool(int64(conns))           // 当前 DC，conns 条连接，无授权转移
    if err != nil { return nil, err }
    waiter := floodwait.NewSimpleWaiter()
    return poolInvoker{Invoker: waiter.Handle(p), closer: p.Close}, nil
}
// 用法：upApi := gtg.NewClient(inv) 喂给 uploader → 200MB 上传 11.5s = 145Mbps

注意：上传(saveFilePart)随连接数线性放大(8 连接=145Mbps)；下载(getFile)单账号则封顶（见 §7），上下行不对称。

5 · 多账号并行分块下载（突破单账号上限）

单账号下载有 per-account 限速。关键实验：4 个账号同时下载同一文件，吞吐会叠加（86 Mbps）还是平分？ → 叠加。说明限速是 per-account 而非 per-IP（上传 145Mbps 也证明服务器 IP 本身不是瓶颈）。

于是把一个文件的分块分发给 N 个账号、共享工作队列（快账号多抓）、块级重试、合并落盘 —— 这就是老项目 TDLID 的真正提速点（单 IP 多账号即可，不需要文档里吹的那套）。

// 共享队列把 [0,num) 块分给所有源(每源=一个账号的无ratelimit连接+该文件对该账号的位置)
func downloadAcrossSources(ctx, sources []dlSource, size int64, dest string, chunk, perAcct int) error {
    idxCh := make(chan int, num); for i:=0;i<num;i++{ idxCh<-i }; close(idxCh)
    for _, s := range sources {            // N 个账号
        for k:=0;k<perAcct;k++ {            // 每账号 perAcct(=4) 个 worker
            go func(s dlSource){
                for idx := range idxCh {    // 抢块：快账号自然多抓
                    off := int64(idx)*int64(chunk)
                    for attempt:=0; attempt<6; attempt++ {   // 块级重试，扛 FLOOD/断连
                        r,e := s.api.UploadGetFile(ctx,&tg.UploadGetFileRequest{Location:s.loc,Offset:off,Limit:chunk})
                        if e!=nil { time.Sleep(...); continue }
                        if fr,ok:=r.(*tg.UploadFile);ok { f.WriteAt(fr.Bytes,off); break }
                        if _,ok:=r.(*tg.UploadFileCDNRedirect);ok { return errCDNRedirect } // 见 §8
                    }
                }
            }(s)
        }
    }
}

实测：27 账号 1GB = 547 Mbps，400MB = 452–468 Mbps；接进备份引擎端到端(下载+连接池上传)463 Mbps，post 真送达。

6 · home-DC / file-DC 致命坑

多账号下载第一版只有 7/27 个账号出力(161Mbps)。原因：channels.getMessages(取每账号自己的 file_reference)是"账号元数据" RPC，必须走账号的 home DC 主连接；而 getFile 走文件所在 DC 的连接池。早期两者都走了文件 DC 的池 → 跨 DC 的账号(home≠file)取不到消息被跳过。修正后 27/27 全出力。

// 每个账号缓存两套句柄：
//   rawApi  = 账号主连接(home DC)  → GetChannelMessage 取 file_reference
//   poolApi = 文件 DC 的连接池      → 仅 getFile 分块
msg := tg.GetChannelMessage(ctx, a.rawApi, a.channel, msgID)   // ← home DC
src := dlSource{ api: a.poolApi, loc: extract(msg).Location }  // ← file DC 池

另外：每文件并发取 file_reference（gather 0.3s）、连接池每趟建一次全趟复用，否则逐文件建连会拖慢吞吐。

7 · 单账号 origin 硬上限 ~49

对冷门/自传/私有文件(origin 单存储节点)，单账号无论怎么调都封顶 ~49 Mbps：

连接池workerMbps
11613–27
41649（甜点）
8 / 16 / 2416撞墙(engine closed / 块重试失败)
432 / 48 / 64撞墙

连接数 >4~7 或单连接 worker >16，Telegram 直接关连接。这是 per-account 的硬限，新号、老号、Premium 都一样。突破只能靠多账号(§5)或 CDN(§8)。

8 · CDN：单会员 315 / 热门源多号 706

转折点：同样是单账号，下载一个热门公开频道里的 325MB 文件 = 315 Mbps（不是 49！）。差异在于热门文件被 Telegram 分发到 CDN / 多节点，而自传冷门文件只在单节点。

要吃到 CDN 必须用 gotd 的"全 client"下载器（downloader.NewDownloader().Download(api, loc)，api 是完整 telegram.Client.API()）——它的 cdn.go 自动处理 upload.fileCdnRedirect → 连 CDN DC → getCdnFile → 解密。DC 绑定的连接池 / 手写 UploadGetFile 都做不到（池只在一个 DC；手写收到 redirect 当成非 UploadFile → 块失败）。

单会员 CDN 的五要素：

① CDN 源热门频道文件(gotd 自动识别 redirect)
② gotd 全 client 下载器非连接池、非手写
③ rate_per_min=0否则被自家中间件掐到 16Mbps
④ 同 DC 会员号跨 DC 仅 ~45
⑤ 线程 ≤ 8>8 撞墙

但更意外的是：对热门源，多账号并行(手写 getFile)也能高速且更快 —— 30 账号 = 706 Mbps(> 单会员 CDN 315)。说明这些热门文件的 getFile 直接从分发节点返回普通 UploadFile(不强制 CDN redirect)，多账号叠加更猛。纯 CDN-redirect 文件才会让手写失败、必须回退单会员全 client。

文件类型多账号并行单会员 gotd全client选谁
冷门/自传(origin)46349多账号
热门(分发节点)706315多账号
纯 CDN-redirect手写失败315单会员

9 · 网速计数器的单位陷阱

仪表盘"看着只有 80"而实测下载 463，一度以为计数器漏采。直接对比网卡 ens7 的 rx_bytes 增量：325MB 文件 → +328MB，流量确在网卡、计数没漏。真相是单位：后端 rx_bps 字段其实是字节/秒，而我们所有测速口径是 Mbps(比特/秒)，差 8 倍。34 MB/s = 272 Mbps 完全对得上。修复：采样 3s→1s（突发也采到）、仪表盘改显 Mbps。

教训：带宽单位务必统一标注 b/B、bit/byte。字段名 rx_bps 存字节是最坑的命名。

10 · 下载调度联动器（最终架构）

把上面所有结论收敛成一个调度器：下载某频道媒体时自动选最快路径。

下载一条媒体：
 ├─ 统计「空闲账号」(在线 且 未在执行别的下载/上传任务)
 ├─ 建多账号下载池(只用空闲号，占用它们)
 ├─ 估算 estMulti = N空闲 × ~17Mbps  vs  CDN单会员 ~300Mbps
 ├─ 路径A 多账号并行分块(origin/热门最快)：
 │     成功 → 用之(冷门463 / 热门706)
 │     收到 CDN redirect(errCDNRedirect) → 转路径B
 └─ 路径B 单会员·同DC·gotd全client(自动解 CDN + 跨DC迁移)：
       pickSameDCPremium(同DC+会员+rate0+空闲) → ~315；无则用执行号兜底

账号忙闲用计数 map 跟踪(acctAcquire/Release)，建池只挑 idleOnlineIDs，关池释放。这样多个备份/下载任务并行时不会互抢同一批账号。

func (w *Worker) smartDownloadMedia(ctx, userID, mdl, fallbackApi, msg, media, dest, threads) (plan string, bytes int64, err error) {
    n := len(mdl.accts)
    if n >= 2 && float64(n)*17 >= 49 {        // 号够多 → 先多账号并行
        b, e := w.mdlDownload(ctx, mdl, msg.ID, dest, 1024, 4)
        if e == nil { return fmt.Sprintf("multi-%dacc", n), b, nil }
        if !errors.Is(e, errCDNRedirect) { /* 真失败,继续兜底 */ }
    }
    // CDN / 无多号 / 多号失败 → 单会员同DC全client
    return w.singleClientDownload(ctx, userID, fallbackApi, media, dest, min(threads,8))
}

实测 @热门源 325MB → 调度自动选 multi-30acc = 706 Mbps。

10.5 · 私有源·加入协调器

公开源所有账号都能 resolve，私有频道却各账号 access_hash 不同，要多账号并行得让每个号"是成员 + 有自己的 hash"。两步：

  • 判成员 / 取本账号 hash：messages.checkChatInvite(hash) —— 已加入返回 ChatInviteAlready{Chat: Channel}（Channel 带本账号 access_hash）；未加入返回预览。
  • 加入：messages.importChatInvite(hash)，返回 updates 含本账号视角的 Channel。

邀请链接从哪来：① 源链接本身就是邀请链接 ② owned_channels 缓存的 invite_link ③ 频道主/管理员账号现 exportChatInvite 生成（并回写缓存）。加入务必限速（每号间隔 2s）+ 本轮限量（防风控）。

for id := range idleOnlineIDs {                       // 只挑空闲号
    ch, isMember := tg.MemberViaInvite(rawApi[id], hash)   // checkChatInvite
    if !isMember {
        ch = tg.JoinViaInvite(rawApi[id], hash)           // importChatInvite
        sleep(2 * time.Second)                            // 加入限速
    }
    // ch 带【该账号】的 hash → 用它 GetChannelMessage 取 file_reference，再喂多账号分块
    sources = append(sources, dlSource{api: dcPool(id), loc: locOf(ch, msgID)})
}

实测：自建私密频道 200MB，调度器自动协调 12 个空闲号通过邀请链接加入 → 13 号并行 = 239 Mbps。私有源从此也能吃满多账号并行。

12 · 踩坑清单

  • per-account ≠ per-IP：单 IP 上多账号下载吞吐叠加，是多账号方案成立的前提。
  • 跨 DC 下载很慢：单账号下载务必挑"home DC = 文件 DC"的号；跨 DC 从 45 起步。
  • 连接池解不了 CDN：CDN 文件只能用完整 client 的下载器；池/手写会 context canceled / 块失败。
  • 导入的无代理账号默认不自动上线：claimLoop 只认 need_relogin / 有代理 / 之前上过线(tg_user_id≠0)。UPDATE accounts SET state='need_relogin' 即可(有效 session → 秒上线免验证码)。
  • 会员号要 rate_per_min=0，否则 CDN 单会员被自家中间件掐到 16Mbps。
  • 部署二进制被截断：旧 worker 在跑高并发下载会吃满上行 → scp 新二进制传一半 → systemd "Text file busy"/203 崩溃循环。部署脚本必须先 systemctl stop + 传完用 stat -c%s 校验大小再 mv。
  • 大流量瞬时 502：下载突发期间 API 偶发 502，登录/轮询要带重试。长任务用"发布即返回 req_id + 轮询取结果"，别长挂 HTTP 连接。
  • PowerShell 坑：大负数 int64 被当 double → 科学计数 → Go json 解析失败(JSON 手拼字符串)；含 () 的 SQL 经 plink→bash 易被解析(写文件 psql -f)。
  • 网速单位：字段叫 *_bps 却存字节/秒，差 8 倍，务必统一。

13 · 给同类产品的 checklist

  1. 必做 文件块传输走无 ratelimit 连接(保留 floodwait)；ratelimit 只套控制面。
  2. 必做 能引用秒传(sendMedia 引文件)就别下载 —— 0 带宽。
  3. 下载 准备两条路：多账号并行分块(冷门/私有源)+ gotd 全 client(CDN/热门源)。
  4. 下载 多账号：home-DC 取消息、file-DC 取块；共享队列；块级重试；池每趟建一次。
  5. 下载 单会员路径选同 DC Premium、rate_per_min=0、线程 ≤8。
  6. 上传 用当前 DC 连接池(8 连接)→ ~145Mbps。
  7. 调度 跟踪账号忙闲，只调度空闲号；多任务并行不互抢。
  8. 运维 部署 stat -c%s 校验大小；长任务 publish+poll；带宽单位统一标 Mbps。

本 cookbook 由"王之矩阵"(Telegram 频道搬运矩阵) 项目的提速实战复盘整理。数据均为单台 VPS + 真实 user 账号实测，可能随 Telegram 策略、地域、文件冷热而变化，请以你自己的实测为准。
